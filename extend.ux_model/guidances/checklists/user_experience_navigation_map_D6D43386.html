Справочная таблица: User-Experience Navigation Map
This checklist contains a list of things to check when reviewing a User-Experience Navigation Map.
Взаимосвязи
Связанные элементы
  • User-Experience Navigation Map
Элементы справочной таблицы
General
  • There is at least one navigation map class diagram that contains the User-Experience Elements and the navigation paths between them.
  • The navigation map is clear and consistent and provides optimal navigation paths for performing system functions
  • The level of detail displayed on the navigation map is consistent with the project-specific guidelines.
  • The number of elements displayed on the navigation map diagram is reasonable (i.e., small enough to view entire diagram on one page).  If not, the diagram should be split into smaller diagrams.
  • All User-Experience Elements have been identified and modeled properly.  For more information, see Checklist: User-Experience Element.
  • All possible navigation paths between the User-Experience Elements have been identified and modeled as associations.  See the "Navigation Associations" section of this checklist for more information.
  • All User-Experience Elements and navigation associations appear on at least one navigation map diagram.
Navigation Associations
  • Every navigation association is named to reflect the reason for the navigation.  For a navigation resulting from a user action, the navigation is named so that it can be easily associated with the initiating user action (i.e., the user action defined for the User-Experience Element that originated the navigation association).
    Remember, every user action does not have to result in a navigation and every navigation is not necessarily the result of a user action.
  • The number of navigation paths is minimal. Ideally, all User-Experience Elements should be "navigate-able to" from a main screen, resulting in a maximum navigation length of two. There should not be any navigation path lengths greater than three.
  • <<input form>> classes are only the source/initiator of a navigation association, not the destination/recipient.  Navigation is always to a <<screen>> class (however, the <<screen>> class my contain an <<input form>> class).
  • Every  <<input form>> class has at least two navigation paths originating from it - a "submitted information valid" navigation path and "submitted information not valid" navigation path.
