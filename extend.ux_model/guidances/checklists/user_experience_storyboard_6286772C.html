Справочная таблица: User-Experience Storyboard
This checklist contains a list of things to check when reviewing a User-Experience Storyboard.
Взаимосвязи
Связанные элементы
  • User-Experience Storyboard
Элементы справочной таблицы
General
  • For each Use Case with a user-experience aspect, there is a User-Experience Storyboard.  The User-Experience Storyboard is modeled as a UML collaboration instance with the stereotype, <<storyboard>>.
  • If a Storyboard exists that describes the same Use Case, then the User-Experience Storyboard is consistent with that Storyboard.
  • If explicit traceability is required, a realization relationship exists from the User-Experience Storyboard to the Use Case it realizes. Alternatively, the same name could be used for the User-Experience Storyboard and the Use Case.
  • The diagrams describing the User-Experience Storyboards are readable and suit their purpose.  See "Participant Diagrams" and "Flow Diagrams" sections for more information.
  • The relationships, such as communicates-associations, include- and extend-relationships, of the corresponding use case in the Use-Case Model are handled correctly within the User-Experience Storyboard.
  • Usability requirements that effect the User-Experience Storyboard are referenced from the User-Experience Storyboard.
Flows of Events
  • The flows of events are readable and suit their purpose (in many cases, the flows of events are captured as the script in the flow diagrams).  If the flows of events are documented in a separate document, the User-Experience Storyboard that the flows of events are associated with is clearly indicated.
  • The flows of events are consistent with the flows of events of the Use Case being realized by the User-Experience Storyboard.
  • The flows of events are consistent with the associated flow diagrams.
  • The flows of events describe how the user interacts with and navigates through the User-Experience Elements while performing the Use Case.  The details concerning what the system does to process the user's request is described in the Use-Case Realizations for the Use Case being realized by the User-Experience Storyboard.
  • The flows of events only specify the sequences that are mandatory in the Use Case.  Specific sequences are not documented as mandatory, where the actual order of operations is determined by the user.
Flow Diagrams
  • There is (at least one) flow diagram for each User-Experience Storyboard flow of events (one for the basic flow and one for each of the alternate flows).
  • All messages on the flow diagrams are be associated with an action (either user or environment) defined on the receiving User-Experience Element.
  • The flow diagrams are consistent with the associated flows of events.
  • In the flow diagrams, only the interactions between the user and the User-Experience Elements and the interactions between the User-Experience Elements themselves are modeled.  No non-presentation elements are included.
Participants Diagram
  • There is at least one participants class diagram for the User-Experience Storyboard.
  • The displayed navigation associations are consistent with the interactions described in the flow diagrams.  For more information, see the "Diagram Consistency Rules" section of this checklist.
  • The navigation associations are consistent with the navigation associations shown in the User-Experience Model Navigation Map. For a specific navigation checklist, see the "Navigation Associations" section of Checklist: User-Experience Navigation Map.
  • The level of detail displayed on the participants diagram is consistent with what is described in the project-specific guidelines.
Diagram Consistency Rules

The following are the consistency rules that should be maintained between the User-Experience Storyboard flow diagrams and participants diagram:

  • For every User-Experience Element instance on the flow diagrams, there should be a class on the participants diagram.
  • For every message in the flow diagram, a supporting navigation association should be defined between the associated classes.  For example, if ObjectA:ClassA sends a message to ObjectB:ClassB, then a navigation association must exist from ClassA to ClassB.
