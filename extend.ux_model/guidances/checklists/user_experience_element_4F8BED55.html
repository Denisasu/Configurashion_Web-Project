Справочная таблица: User-Experience Element
This checklist contains a list of things to check when reviewing a User-Experience Element.
Взаимосвязи
Связанные элементы
  • User-Experience Element
Элементы справочной таблицы
General
  • The User-Experience Element has a name that reflects its responsibility.  Names must be unique, and no two elements should have synonymous names.
  • There should be a brief description defined for the element itself, as well as for any dynamic content and user actions.  Remember, the User-Experience Model is intended to be contract between the user-experience and the engineering teams.  Thus, it is important that it is unambiguously documented.
  • The properties of the User-Experience Element, including its actions, navigation associations, and content, are justified and consistent with each other.
  • The necessary navigation associations have been defined between User-Experience Elements, and the navigation associations are consistent.  See the "Navigation Associations" section of this checklist for a specific navigation association checklist.
  • The element's Special Requirements are readable and suit their purpose.
Screens
  • The User-Experience Element has a <<screen>> stereotype.
  • The definition of each screen includes the dynamic content to appear on the element (modeled as attributes), as well as any user actions or environment actions that can be invoked on the element (modeled as operations).
  • If the screen contains input fields, <<input form>> class(es) have been modeled (a single screen may contain multiple input forms).  There is a composition relationship from the screen to the contained input forms.
    For more information, see the "Input Forms" section of this checklist.
  • If messages are to be displayed on a screen, the necessary dynamic content attributes have been defined, as well as the environment actions that display the messages.  For more information on modeling screen messages, see Guideline: User-Experience Element, Modeling Screen Messages.
  • The screen participates in at least one User-Experience Storyboard.
  • The screen fulfills the requirements made on it from the use-case realizations and storyboards in which it participates.
Input forms
  • The User-Experience Element has a <<input form>> stereotype.
  • Every input form is part of a screen. The relationship between screens and their input forms is modeled as a composition relationship from the <<screen>> class to the contained <<input form>> class(es). 
    See the "Screens" section of this checklist for more information.
  • The name of the input form should reflect the data that is being entered on the form, and it is recommended that "form" be included as the last word in the class name (e.g., "account info form").
  • The definition of each input form includes the input fields (modeled as attributes), as well as any user actions that can be invoked on the element (modeled as operations).
  • There is at least two user actions defined for the input form -- an action to submit the form content and an action to cancel the submission. Another useful action is an action to clear the form of any entered information.
