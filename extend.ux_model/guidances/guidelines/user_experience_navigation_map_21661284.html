Рекомендация: User-Experience Navigation Map
These guidelines provide modeling conventions and recommendations for representing the User-Experience Navigation Map.

Основное описание

The User-Experience Navigation Map is represented in a UML class diagram containing the User-Experience Elements and the key navigation paths between them.  These associations don't represent every possible path, just the ones captured in the Storyboards' flows of events. This distinction is important because the User-Experience Model needs to remain detached from the details of the software architecture as much as possible, since the intent is not to present just another view of the detailed Design Model. For information on how to model the User-Experience Elements, see Guideline: User-Experience Element.

There is one User-Experience Navigation Map per User-Experience Model (the navigation map is essentially a combination of all of the User-Experience Storyboard participant diagrams). If the total number of User-Experience Elements is small enough to permit them to realistically appear in a single diagram (say, less than 30 elements), then one diagram can be used. If there are too many User-Experience Elements, then the User-Experience Navigation Map should be expressed as a series of diagrams, with each diagram focusing on a related set of information. Some possibilities for separate User-Experience Navigation Map diagrams include:

  • Actor-focused class diagrams.  Each would contain the User-Experience Elements that a particular actor might view.

  • Function-focused class diagrams. Each would contain the User-Experience Elements that are accessed when performing a particular set of application functions (like account maintenance).
    Note: It may be tempting to create Use-Case-focused diagrams.  However, such diagrams would be redundant with the User-Experience Storyboard participant diagrams.

  • Architecturally significant User-Experience Elements.  The most important User-Experience Elements and their navigation paths. Diagrams of this type can function as a summary of the User-Experience Model, and can be of great help in, for example, a review of the model, or in a prioritization of which parts of the model will be focused upon in the current iteration.

To reduce clutter on the User-Experience Navigation Map diagrams, just include the User-Experience Elements, their names, and their navigation associations.  Do not include the User-Experience Elements dynamic content (attributes) or actions (operations).  That level of detail is reserved for the Participant diagrams defined within the User-Experience Storyboards, where the scope of the diagram is a little more manageable (i.e., a single Use Case as opposed to an entire application).  The User-Experience Navigation Map is meant to be a summation of these individual diagrams and provide a "big picture" of the structure of the User-Experience Model.

The decision to what level of detail is appropriate for User-Experience Navigation Map diagrams will depend upon the individual teams building the model. Your project may want to define exactly what should be developed by your teams. These guidelines are meant to provide a starting point for a development team to tailor their own guidelines.  Once decided upon, the guidelines for the project should be captured in the project-specific guidelines.

For more information on identifying and modeling navigation for User-Experience Elements, see Guideline: User-Experience Element, Navigation.
