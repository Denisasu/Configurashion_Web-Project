Артефакт: User-Experience Model
This work product describes the user-experience elements of the system (the screens and input forms), the dynamic content that appears on the elements, and how the user navigates through the elements to execute the system functionality.
Домены: Анализ и проектирование
Типы рабочих продуктов: Модель
Назначение

The User-Experience Model supports the reasoning about, and the enhancement of, the usability of the system's user interface. It provides a platform for communication between the technology-focused team members and the creative team members. The User-Experience Model is a very important system model if an extensive part of the system is dependent on the user experience, and if usability is being emphasized. 

The User-Experience Model provides value throughout the software lifecycle.  In the early phases, the User-Experience Model provides the contract between the presentation and the business aspects of the system.  In the later phases, the User-Experience Model provides a baseline from which to assess the usability of the evolving system.  The User-Experience Model is intended to be maintained and represent the user experience that (initially) is to be implemented and (finally) is implemented.

Описание
Основное описание

The User-Experience Model:

  • Rigorously and unambiguously describes the User-Experience Elements, the content of those elements, and the ways in which the user may interact with these elements. Each User-Experience Element defines content and user actions that have a straightforward mapping to the corresponding construct in the user interface.
  • Captures the essence of the architecturally significant User-Experience Elements and provides an appropriate view of those elements to the Software Architects, Designers and Implementers of the rest of the system.
  • Provides a high-level view of static user interface element relationships such as window containment hierarchies and other associations between elements in the user interface.
  • Provides a high-level view of dynamic user interface element relationships such as window navigation paths.

There are multiple benefits for modeling the user experience in the same "language" (i.e., UML) as the modeling of the application.  Developing a UML model of the user interface is very worthwhile, since many user interfaces created today are object-oriented, and the trend is towards even more object orientation, because of its benefits with regard to natural and efficient use. Although many of the user-interface design decisions are best made during prototyping and rapid development of the user interface itself, reasoning about the structure and usability requirements on the user interface is natural to do in terms of an object model.

The User-Experience Model is used by the following roles:

  • user-interface designers use it to prototype the user interface in order to validate the User-Experience Model.
  • designers of the presentation logic design elements, use it to understand the User-Experience Element's roles in the system functionality and how the User-Experience Elements are used together. The also use it to design and implement the presentation logic design elements such as web pages, and applets that will implement the User-Experience Elements.
  • designers of the business logic design elements, use it to understand what business content is required by the User-Experience Elements and must be provided by the business logic.
  • those who design the next version of the system use it to understand how the system carries out the flow of events in terms of User-Experience Elements. For example, a change may affect a limited number of Use Cases, in which case the designers need to see the realization of their flow of events.
  • those who test the system's Use Cases.

Ключевые условия

The User-Experience Model is most effective when it is developed early in the lifecycle since the user-experience is a critical part of the overall system architecture and has a direct effect on the software architecture. Also, the process of developing the User-Experience Model may uncover vague or missing requirements.

To develop a practical User-Experience Model, at least a brief outline of the Use Cases must exist.  Detailed Use Cases are not required, but there must be some understanding of the functionality the system is to perform, as well as the intended look-and-feel of the system. The User-Experience Model can be developed at the same time the Design Model (or the Analysis Model, if there is one) is being developed (see Activity: Analyze Behavior).  

Traceability is then maintained throughout the software lifecycle between the User-Experience Model and the other system models.  For more information on this traceability, see Guidelines: User-Experience Model, Traceability Between the User-Experience Model and Other System Models.

For guidelines on developing the User-Experience Model, see Guideline: User-Experience Model.
Доводка
Причины для отказа

A User-Experience Model is not needed if the user-experience aspect of the system is well known, of low risk, or of minor importance (i.e., not considered critical to the success of the system).  In such cases, the relevant aspects of the user-experience can be captured in the Analysis Model and/or Design Model (see Task: Use-Case Analysis, specifically the identification of boundary Analysis Classes).

Опции представленияUML Representation: Model, stereotyped as <<user-experience model>>

If it is decided that a User-Experience Model is to be maintained, then the following must be decided:

  • What properties to include.
  • If any extensions to the Unified Modeling Language (UML) are needed.  For example, your project may require additional stereotypes. 
  • The level of formality that should be applied.
  • If the User-Experience Model will be elaborated to become executable and testable.

When creating the User-Experience Model, the level of detail should be appropriate to the intended audience (the User-Experience Model needs to be understood by the creative/presentation team, as well as the business logic team).  If the audience is unfamiliar with UML, and is unlikely to adopt it, then it is best to eliminate some of the modeling details.
